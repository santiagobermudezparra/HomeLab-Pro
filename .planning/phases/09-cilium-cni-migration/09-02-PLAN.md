<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Install Cilium via helm and wait for nodes Ready</name>
  <read_first>
    - .planning/phases/09-cilium-cni-migration/09-RESEARCH.md (Step 4 and Step 5 sections)
  </read_first>
  <files>none (imperative cluster operations)</files>
  <action>
    Execute the following steps IN ORDER. The cluster currently has no CNI (from Plan 01).

    STEP 1 — Add cilium helm repo and install Cilium 1.16.19:
    ```bash
    helm repo add cilium https://helm.cilium.io/
    helm repo update

    helm install cilium cilium/cilium \
      --version 1.16.19 \
      --namespace kube-system \
      --set k8sServiceHost=127.0.0.1 \
      --set k8sServicePort=6444 \
      --set "ipam.operator.clusterPoolIPv4PodCIDRList={10.42.0.0/16}" \
      --set operator.replicas=1 \
      --set hubble.enabled=true \
      --set hubble.relay.enabled=true \
      --set hubble.ui.enabled=true
    ```

    STEP 2 — Wait for Cilium DaemonSet and Operator to be ready:
    ```bash
    kubectl rollout status daemonset/cilium -n kube-system --timeout=5m
    kubectl rollout status deployment/cilium-operator -n kube-system --timeout=5m
    ```

    If rollout status times out: check pod logs with `kubectl logs -n kube-system -l k8s-app=cilium --tail=50`
    Common failure: flannel.1 not deleted → check `ip link show flannel.1` (must not exist from Plan 01 Task 2)

    STEP 3 — Verify all nodes return Ready:
    ```bash
    kubectl get nodes
    ```
    Keep running until all 3 show Ready. May take up to 2 minutes after Cilium pods Running.
    If a node is stuck NotReady after 5 minutes:
    - `kubectl describe node <node-name>` — look for CNI errors
    - `kubectl logs -n kube-system -l k8s-app=cilium` — look for startup errors

    STEP 4 — Install cilium CLI:
    ```bash
    CILIUM_CLI_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/cilium-cli/main/stable.txt)
    curl -L --remote-name-all \
      https://github.com/cilium/cilium-cli/releases/download/${CILIUM_CLI_VERSION}/cilium-linux-amd64.tar.gz
    tar xzvf cilium-linux-amd64.tar.gz
    sudo mv cilium /usr/local/bin/cilium
    rm -f cilium-linux-amd64.tar.gz
    ```

    STEP 5 — Verify cilium status:
    ```bash
    cilium status
    ```
    Expected: All nodes show green/OK. "Cilium" row shows "OK". "Hubble Relay" row shows "OK".
    If any node shows an error: do NOT proceed — diagnose first.
  </action>
  <verify>
    ```bash
    kubectl get nodes
    kubectl get pods -n kube-system | grep cilium | grep Running
    cilium status
    kubectl get pods -n kube-system | grep hubble
    ```
  </verify>
  <done>
    All 3 nodes Ready. cilium DaemonSet pods Running (one per node). cilium-operator Running.
    hubble-relay and hubble-ui pods Running. cilium status exits 0 with OK output.
  </done>
  <resume-signal>Type "cilium healthy, nodes Ready" after all checks pass, or describe errors encountered</resume-signal>
</task>

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Restart all pods to use Cilium networking</name>
  <read_first>
    - .planning/phases/09-cilium-cni-migration/09-RESEARCH.md (Step 6 section)
  </read_first>
  <files>none (imperative cluster operations)</files>
  <action>
    Pods created under Flannel have stale network namespaces referencing old veth pairs.
    They must be restarted to get Cilium networking. This is a rolling restart — apps will have
    brief downtime as each pod restarts.

    STEP 1 — Rolling restart of all Deployments:
    ```bash
    kubectl get deployments --all-namespaces -o json | \
      jq -r '.items[] | .metadata.namespace + " " + .metadata.name' | \
      xargs -n2 kubectl rollout restart deployment -n
    ```

    STEP 2 — Rolling restart of all StatefulSets:
    ```bash
    kubectl get statefulsets --all-namespaces -o json | \
      jq -r '.items[] | .metadata.namespace + " " + .metadata.name' | \
      xargs -n2 kubectl rollout restart statefulset -n
    ```

    STEP 3 — Rolling restart of all DaemonSets except cilium (already using Cilium networking):
    ```bash
    kubectl get daemonsets --all-namespaces -o json | \
      jq -r '.items[] | select(.metadata.name != "cilium" and .metadata.name != "cilium-envoy") | .metadata.namespace + " " + .metadata.name' | \
      xargs -n2 kubectl rollout restart daemonset -n
    ```

    STEP 4 — Wait for all pods to stabilize:
    ```bash
    sleep 60
    kubectl get pods --all-namespaces | grep -v Running | grep -v Completed
    ```
    Expected: empty output (no pods in Error, CrashLoopBackoff, or Pending).
    If any pods are in CrashLoopBackoff:
    ```bash
    kubectl logs -n <namespace> <pod-name> --previous
    ```

    STEP 5 — Verify flux-system NetworkPolicies intact:
    ```bash
    kubectl get networkpolicies -n flux-system
    ```
    Expected: 3 policies — allow-egress, allow-scraping, allow-webhooks

    STEP 6 — Verify CNPG clusters healthy:
    ```bash
    kubectl get clusters.postgresql.cnpg.io -A
    ```
    Expected: Both linkding and n8n clusters show "Cluster in healthy state" or Ready instances.

    STEP 7 — Verify Cloudflare tunnel pods running:
    ```bash
    kubectl get pods -A -l app=cloudflared
    ```
    Expected: All cloudflared pods Running.
  </action>
  <verify>
    ```bash
    kubectl get pods --all-namespaces | grep -v Running | grep -v Completed
    kubectl get networkpolicies -n flux-system
    kubectl get clusters.postgresql.cnpg.io -A
    kubectl get pods -A -l app=cloudflared
    cilium status
    ```
  </verify>
  <done>
    No pods in non-Running state.
    flux-system has 3 NetworkPolicies (allow-egress, allow-scraping, allow-webhooks).
    Both CNPG clusters healthy.
    All cloudflared pods Running.
    cilium status still OK.
  </done>
  <resume-signal>Type "all pods Running, NetworkPolicies intact" after all checks pass, or describe any pods that failed to restart</resume-signal>
